# Communication Conventions

## The basics
- Plain English only — no tech jargon unless necessary
- Short answers — one or two sentences when possible
- No explaining why something broke or the details of a fix
- No long summaries of what was just done

## What to focus on
- What decision needs to be made
- What was built or changed (one line)
- What's next
- If something is blocked, say so simply and say what's needed

## What to skip
- Technical explanations of errors
- Step-by-step breakdowns of how code works
- Reasons behind implementation choices (unless asked)
- Recapping what was already said

## Asking questions
- One question at a time when possible
- Give options when a decision is needed (not open-ended "what do you want?")
- If a feature has obvious missing pieces, flag them briefly

## Building
- Always ask before building anything new
- Confirm the plan in plain terms before writing code
- If something is unclear, ask — don't guess and build

## Progress updates
- Short bullet or one sentence max
- "Done. Next: X" is enough
